## 主項目：Lab 容器網路隔離

### 問題說明

v2.1 為了避開「nginx 已驗 JWT 放行 + code-server 又彈 password login」死鎖，把 `infrastructure/base-images/code-server/Dockerfile` 設成 `--auth none`。改採「**nginx auth_request 是唯一認證閘**」的設計。

但目前所有 lab 容器都掛在同一個 `ai-platform-net` (bridge network)，造成：

```
┌─ ai-platform-net ──────────────────────────────┐
│                                                │
│   nginx ──auth_request──▶ scheduler            │
│                                                │
│   cs-<user_A_uid> :8080  (--auth none)         │
│   cs-<user_B_uid> :8080  (--auth none)         │
│                                                │
│   ❌ user_A 容器內可直接 curl cs-<B_uid>:8080  │
│      繞過 nginx auth_request                   │
└────────────────────────────────────────────────┘
```

### 攻擊路徑

1. 學生 A 在自己 Lab 終端打開
2. `curl http://cs-<other_uid>:8080/` (cs-xxx 主機名透過 Docker DNS 解析)
3. 因 `--auth none`，code-server 直接回 HTML / 提供檔案讀取 API
4. **可看到別 user 的工作目錄、原始碼**

### 攻擊要件

- 知道目標 user 的 UUID（admin UI 可見、API 列表也可能洩漏）
- 攻擊者願意在自己 lab 容器內主動執行 curl（明顯異常）

### 為什麼 v2.1 不直接做

- **教學平台、低威脅模型**：使用者是學生、老師，不對抗高水準入侵
- 真實攻擊需要明顯異常行為（容器內 curl 其他容器名稱）
- 工時投入比例不對：完整解 4-6 hr vs 風險暴露程度

### 解法評估

#### 解法 A：nginx + shared secret header + socat sidecar

- code-server 不仗 `--auth none`，回到 `--auth password`
- 在 code-server 容器旁加 socat sidecar，攔截 :8080 流量
- nginx 在 auth_request 通過後，proxy 給 code-server 時帶一個 `X-Auth-Secret` header
- socat sidecar 檢查 header 是否符合 → 才轉給 code-server
- code-server 內部仍走 password，但 nginx 把 password 在 cookie 裡塞給瀏覽器

**缺點**：增加一個 sidecar 容器，維運複雜度上升。code-server 原生不支援 header-based auth。

#### 解法 B：per-user docker network（**推薦**）

每位使用者一個獨立 docker network，nginx 動態加入該 network。

```
┌─ ai-platform-net (基礎設施) ──────┐
│   nginx, scheduler                │
└────────────┬──────────────────────┘
             │ nginx 動態 connect
             ▼
┌─ lab-net-<user_A_uid> (per-user) ─┐
│   cs-<user_A_uid> :8080 (內部隔離)│
└───────────────────────────────────┘

┌─ lab-net-<user_B_uid> (per-user) ─┐
│   cs-<user_B_uid> :8080 (內部隔離)│
└───────────────────────────────────┘

user_A 的 lab 容器無法看到 user_B 的 network
nginx 在兩個 net 都有 leg，仍可 proxy
```

**實作藍圖**：

1. **新檔** `job-scheduler/app/services/network_manager.py`
   - `def ensure_user_network(user_id: str) -> Network`：呼叫 docker API `networks.create(name=f"lab-net-{user_id}", driver="bridge", internal=False)`（internal=True 則無 outbound 網路，依需求決定）
   - `def connect_nginx_to_network(network: Network)`：用 `container.connect(NGINX_CONTAINER_ID)`
   - `def cleanup_orphan_networks()`：列出 `lab-net-*` 找沒有對應 LabSession 的，斷開 nginx + 刪除

2. **改 `lab_manager.py`** `_codeserver_lifecycle.start()` 第 142 行附近：
   ```python
   # 原本
   network="ai-platform-net",
   
   # 改成
   user_network = network_manager.ensure_user_network(user_id)
   network_manager.connect_nginx_to_network(user_network)
   ...
   network=f"lab-net-{user_id}",
   ```

3. **改 `lab_manager.py`** `stop_session()`：
   ```python
   # 容器移除後
   network_manager.disconnect_and_cleanup(user_id)
   ```

4. **scheduler 啟動 hook**：呼叫 `network_manager.cleanup_orphan_networks()` 清理重啟前殘留

5. **nginx 不用動**：`proxy_pass http://cs-<uid>:8080` 仍可用（nginx 加入 user network 後 DNS 解析依然有效）

**風險點**：

- **時序**：nginx 加入 network → 立即 proxy 之間有微小延遲，可能需要 retry
- **Orphan 清理**：容器死亡但 network 沒清掉
- **網路上限**：docker bridge driver 預設限制 30 個網路，若 user 數 >30 需切 swarm overlay

**工時估算**：4-6 hr（含測試 + 文件）

**遷移風險**：低 — 不動 nginx config / 前端 / database schema / API；改動全在 lab_manager + 新檔 network_manager

## 其他 v2.2 候選項目（待排）

### Lab 容器 secrets 隔離稽核

目前 secrets 注入容器後即明文，container exec 可讀。考慮：
- 改用 docker secrets（會 mount 到 `/run/secrets/`，較難 dump 但 root 仍可讀）
- 或加 audit log：誰何時讀取了 secrets

### SSO group claim → role 自動對應

v2.1 OIDC 登入預設 `role=student`。v2.2 計畫從 Microsoft Entra ID 的 group claim 自動 promote：
- 該 group 屬於「教師」→ `role=teacher`
- 不在任何已知 group → 維持 `student`
- admin 必須仍由現有 admin 手動提權（防 group 設定錯誤導致權限漏洞）

### Single Logout (SLO)

OIDC 規範的 RP-initiated logout：本平台登出時，順便登出 Microsoft session。目前只清本機 localStorage + cookie。

### id_token jwks 簽章驗證

v2.1 OIDC client 沒驗 id_token 的 RSA 簽章（直接信任 token endpoint 回來的內容，因為走 HTTPS）。v2.2 加 jwks 公鑰驗證 + 結果 cache。

### `/api/v1/jobs/` redirect 修法的反向遷移路徑

v2.1 採 `redirect_slashes=False`，未來若想恢復 FastAPI 預設行為，需要：
- 確認所有 router `@router.get("")` 改成 `@router.get("/")` 並驗證所有前端調用點
- nginx regex location 改回 prefix location（trailing slash 統一）
