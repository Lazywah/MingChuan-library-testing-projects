# 07 - 工具組件統整 | Tools & Components Integration

本文件用於統整 AI 訓練平台中所使用的各項核心工具與中介軟體，方便開發者快速查閱技術細節、完成狀態與使用方式。

---

## 1. Open WebUI
- **所在檔案**：`docker-compose.ai-models.yml` (服務名稱: `open-webui`)
- **相關工具**：Ollama, Portkey, Dartmouth Token Tracking
- **使用狀況**：作為平台上層最主要的使用者對話介面，提供類似 ChatGPT 的網頁操作體驗。其透過環境變數 `OLLAMA_API_BASE_URL` 與 Portkey 閘道器對接。
- **完成率**：100% (已整合於 Docker 佈署方案)
- **使用方法**：瀏覽 `http://localhost:3000` 即可登入使用。
- **對應輸入輸出**：
  - **輸入**：使用者對話文字、上傳圖片（多模態分析）。
  - **輸出**：AI 生成回應、歷史對話紀錄。

---

## 2. Portkey (Gateway)
- **所在檔案**：`docker-compose.ai-models.yml` (服務名稱: `portkey`), `./portkey/config.yaml`
- **相關工具**：OpenAI, Anthropic, Google Gemini, Ollama
- **使用狀況**：擔任 LLM API 與模型請求的分流網關。系統所有模型請求皆先經過 Portkey 進行日誌記錄、重試與多供應商路由。
- **完成率**：100% (已實作核心配置與閘道邏輯)
- **使用方法**：API 端點位於 `http://(portkey_ip):8000`；配置設定位於 `./portkey/config.yaml`。
- **對應輸入輸出**：
  - **輸入**：標準 OpenAI 格式的 API 請求。
  - **輸出**：標準化後的各家模型業者回應。

---

## 3. Ollama
- **所在檔案**：`docker-compose.ai-models.yml` (服務名稱: `ollama`)
- **相關工具**：Llama3, Open WebUI, Portkey
- **使用狀況**：本地備援推論引擎。當外部 API 斷開或需運行本地模型時，負責載入 GGUF 模型進行運算。
- **完成率**：100% (已實作於 AI 模塊編排)
- **使用方法**：進入容器執行 `ollama run llama3` 或透過 API `http://localhost:11434` 通訊。
- **對應輸入輸出**：
  - **輸入**：模型標籤、提示詞 (Prompt)。
  - **輸出**：模型推論結果、Token 生存率數據。

---

## 4. Dartmouth Token Tracking (Open WebUI)
- **所在檔案**：外部掛載介面 / Open WebUI Functions 腳本 (預算法/過濾器)
- **相關工具**：Open WebUI, Token Metrics
- **使用狀況**：用於在 Open WebUI 介面中即時顯示每筆對話消耗的 Token 數量，助於使用者監控配額使用情形。
- **完成率**：50% (目前由 AI 平台後端攔截計算，前端 UI 整合待進一步對齊)
- **使用方法**：於 Open WebUI 之「首選項 (Workspace) -> Functions」中啟用相關腳本。
- **對應輸入輸出**：
  - **輸入**：對話 API 的 `usage` 欄位數據。
  - **輸出**：於對話氣泡下方顯示消耗的具體數值 (Input/Output Tokens)。

## 5. JupyterHub
- **所在檔案**：`infrastructure/jupyterhub` (規劃中)
- **相關工具**：JupyterLab, Python, DockerSpawner
- **使用狀況**：為研究員提供一站式、基於網格的數據中心開發環境，支援多使用者獨立的程式撰寫視景。
- **完成率**：0% (Roadmap 項目，待下一階段實作)
- **使用方法**：預計透過 `http://(server_ip):8000/jupyter` 進入。
- **對應輸入輸出**：
  - **輸入**：Python/R/Julia 原始碼、數據集。
  - **輸出**：互動式運算圖表、Jupyter Notebook 檔案 (`.ipynb`)。

---

## 6. Slurm (Workload Manager)
- **所在檔案**：`job-scheduler/app/gpu_client.py` (抽象基底介面)
- **相關工具**：SSH, NVIDIA GPU, Open MPI
- **使用狀況**：高效能運算 (HPC) 資源調度器。負責在大規模 GPU 叢集上管理與調度使用者的訓練任務。
- **完成率**：15% (已實作 `BaseGPUClient` 與工廠模式，支援 mock 模擬，Slurm 客戶端實作中)
- **使用方法**：後端排程器透過 `sbatch` 或 `srun` 與 Slurm 進行通訊。
- **對應輸入輸出**：
  - **輸入**：訓練腳本目錄、要求之 GPU 資源數。
  - **輸出**：任務編號 (Job ID)、排程狀態、節點資源回報。

---

## 7. NVIDIA DCGM (Data Center GPU Manager)
- **所在檔案**：GPU 伺服器底層 / Prometheus Exporter
- **相關工具**：Prometheus, Grafana, Slurm
- **使用狀況**：硬體監控核心。用於採集 GPU 的溫度、負載、功耗及發生 XID 錯誤時的即時攔截。
- **完成率**：0% (待正式伺服器節點上線後，透過 Exporter 整合至儀表板)
- **使用方法**：透過 `dcgmi` 指令集或 HTTP Metrics 端點獲取數據。
- **對應輸入輸出**：
  - **輸入**：GPU 硬體感測器訊號。
  - **輸出**：時序遙測數據 (Telemetry Data)。
