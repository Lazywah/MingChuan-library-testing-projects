# v2.1 Roadmap — AI Base Lab 進階功能

> **狀態**：規劃中（v2.0 上線後再啟動）
> **核心原則**：**架構抽象 + 文件指引**，**不在 v2.0 預留半實作 schema/API**（避免攻擊面）

---

## 為什麼 v2.0 不直接做這些

v2.0 已交付的「Edit + Run 雙軌」（code-server CPU 容器 + GPU 容器任務驅動）能順利支援：

- ✅ 寫完整訓練腳本一次跑完（教學示範、固定範本、學生作業）
- ✅ 推論服務 / 資料前處理
- ✅ Notebook Cell Run（雖然每個 cell 是獨立 Job）

但**研究生的迭代式開發**仍會卡：

- ❌ 大模型互動式 debug（OOM 找 tensor、ipdb 進去看）
- ❌ 高頻迭代實驗（每次 5–10 分鐘 GPU 容器啟動延遲不可接受）
- ❌ 全量微調 7B+ 模型（需要 2–4 張 A100 多卡並行）
- ❌ 變數共享跨 cell（每個 Run 都是新 GPU 容器）

v2.1 解決這些。

---

## v2.1 預定加入的功能

### 1. Jupyter 真 Kernel 模式（最高優先級，~3–5 天）

**動機**：研究生需要「常駐 GPU + 變數共享 + interactive debug」，等同 Colab 體驗。

**設計（不動 v2.0 schema/API）**：

- **新檔**：`job-scheduler/app/services/kernel_manager.py`
  - 實作 `KernelLifecycle(ContainerLifecycle)`，繼承 v2.0 抽象介面
  - 啟動 GPU 容器跑 `jupyter_server` + `ipykernel`，**不掛 `--rm`**
  - 容器名 `kernel-{user_id}`，idle 15 分鐘 evict

- **新檔**：`job-scheduler/app/routers/kernels.py`
  - `POST /api/v1/kernels/start` → 啟動 GPU kernel 容器，回傳 jupyter token + URL
  - `POST /api/v1/kernels/stop`
  - `GET  /api/v1/kernels/status`
  - **不要動** `routers/lab.py`（保持 code-server 端不變）

- **新表**：`kernel_sessions`（不要在 `lab_sessions` 加欄位）
  ```python
  class KernelSession(Base):
      user_id      = Column(String, ForeignKey("users.id"), primary_key=True)
      gpu_id       = Column(Integer)
      container_id = Column(String)
      jupyter_token = Column(String)  # ad-hoc one-time token
      last_activity = Column(DateTime)
      started_at   = Column(DateTime)
  ```

- **VS Code extension 擴充**：`aibase-runner` 加新指令
  - `AI Base: Connect GPU Kernel` → 呼叫 `/api/v1/kernels/start` → 用 VS Code Jupyter 內建的「Specify Jupyter Server」連線
  - **保留** `Run on GPU`（單次 Job）作為 fallback

- **GPU 分配政策**（**v2.1 必須新增**，v2.0 不要動）
  - 新表 `gpu_reservations` 追蹤 interactive 容器佔用
  - WorkerHeartbeat 計算「真實可用 GPU」時排除 reservations
  - batch Job `/take` 端點檢查 reservation 表
  - 或：dedicated kernel pool（從 scheduler_policy.yaml 配置）

**驗收標準**：
1. 研究生 `import torch; model = AutoModel.from_pretrained("7B")` 一次後，後續 cell 可直接用 `model`
2. OOM 後可在 cell 內 `del model; torch.cuda.empty_cache()` 而不重啟 kernel
3. idle 15 分自動 stop；研究生再開時 kernel 已關，但 volume 內檔案仍在

---

### 2. 多 GPU 任務支援（中優先級，~1.5 天）

**動機**：全量微調 7B 模型需 2–4 張 A100；vLLM tensor-parallel 推論。

**設計**：

- **修改既有 schema**（v2.1 自行 ALTER，**v2.0 不要預留**）
  ```sql
  ALTER TABLE training_jobs ADD COLUMN gpus_required INTEGER DEFAULT 1;
  ALTER TABLE training_jobs ADD COLUMN gpu_ids TEXT;  -- JSON list ["0","1"]
  ```

- **修改 `routers/worker.py` `/take` 邏輯**
  - 若 `gpus_required > 1`：只回覆 `available_gpus` 數量足夠的 Worker
  - 一次分配多張 GPU id 給該 job
  - 同時把這些 GPU 標記為佔用

- **修改 `gpu-worker/worker.py`**
  - `docker run --gpus device=0,1,2,3` 多卡
  - busy-set 同時鎖多張卡

- **驗收**：
  1. 提交 `gpus_required=2` 的 job → Worker `/take` 只給有 2+ 張可用的節點
  2. 4 張 A100 節點上能並存 4 個單卡 job 或 2 個雙卡 job
  3. nvidia-smi 顯示所有分配的 GPU 都跑同一個 docker container

---

### 3. Secrets 金鑰輪換（低優先級，~1 天）

**動機**：90 天輪換 KEK 是基本安全週期；v2.0 用單一 KEK。

**設計**：

- **新工具**：`scripts/rotate_secrets_master_key.py`
  - 接受舊 KEK + 新 KEK
  - 把所有 `user_secrets.value_enc` 解密 + 用新 KEK 重新加密
  - 寫入新版 `.env`
  - 不停機進行（用 SQL transaction 分批處理）

- **支援雙 KEK 過渡期**
  - `SECRETS_MASTER_KEY` 為新 key
  - `SECRETS_MASTER_KEY_OLD` 為舊 key（解密 fallback）
  - `secrets_service.py` 解密失敗時嘗試 fallback、解密成功後立即用新 key 重新加密儲存

## v2.0 絕對「不要做」的事

為避免 v2.1 介面變動或攻擊面擴大，**v2.0 嚴禁**：

- ❌ 在 `lab_sessions` 加 `is_interactive` / `mode` 欄位
- ❌ 在 `training_jobs` 加 `gpus_required > 1` schema（即使保留為 nullable）
- ❌ Job API 接受 `reserved` / `kernel_mode` 之類「未使用」參數
- ❌ WorkerHeartbeat 加 reserved GPU 計數欄位
- ❌ 預先建立 `kernel_sessions` 空表

**理由**：未使用欄位若沒 validator，就是攻擊面；若加 validator 變成半實作 bug；v2.1 真做時又得改設計，浪費時間。

## v2.0 已做好的抽象（v2.1 直接重用）

- ✅ `services/lab_manager.py` 用 `ContainerLifecycle` Protocol → v2.1 新增 `KernelLifecycle` 不動既有
- ✅ Per-user volume 設計 → kernel 容器掛同一塊，狀態天然延續
- ✅ JWT + nginx auth_request → 沿用同一機制驗證 kernel 路由
- ✅ Worker heartbeat 的 `pool_type` 欄位 → v2.1 直接用 `interactive` 值區分
- ✅ `admin_actions` audit log → 任何 v2.1 新動作都記錄到同一張表
- ✅ `scheduler_policy.yaml` 的 `pools` 區塊 → kernel pool 設定加在這

## v2.1 完工時機評估

當以下任一條件成立，就應該啟動 v2.1：

1. **研究生抱怨累積到 5 次**：「想 debug 大模型」「Run All 太慢」「想看中間 tensor」
2. **管理員報告**：日常被「OOM 後重提交」的 job 佔掉 30%+ GPU 時間
3. **使用者數成長**：日活 ≥ 30 人，有明顯研究生比重
4. **採購了新硬體**：≥ 8 張 A100，多卡訓練成為日常需求

v2.0 在前面期間已足夠承載 ≤20 人小規模 + 教學/作業場景。
