# 00 — 本機完整部署指南（單機 PC / Windows + Docker Desktop）

> 本文記錄在**單一 PC** 上把整個平台（核心服務 + AI 模型 + GPU 訓練）完整跑起來的
> 所需工具、環境、啟動步驟、測試結果，以及部署過程中發現並修正的問題。
> 撰寫時的實測環境：**Windows 11 + Docker Desktop + NVIDIA RTX 4060（8GB）**。

---

## 1. 所需工具 / 環境

| 項目 | 需求 | 備註 |
|---|---|---|
| OS | Windows 10/11（或 Ubuntu 22.04+） | 本指南以 Windows + Docker Desktop 為主 |
| Docker | Docker Desktop（含 Compose v2） | 實測 Docker 28.x / Compose v2.39 |
| WSL2 | 啟用（Windows） | Docker Desktop 勾「Use WSL 2 based engine」|
| GPU（選用） | NVIDIA GPU + 驅動 | 沒有 GPU 也能跑（Ollama/訓練改 CPU，較慢）|
| NVIDIA GPU passthrough | `docker run --rm --gpus all nvidia/cuda:12.3.0-base-ubuntu22.04 nvidia-smi` 能看到 GPU | Docker Desktop 通常自動具備 |
| 磁碟空間 | ≥ 60–80 GB | base images ~20GB×數個 + llama3 4.7GB + open-webui 等 |
| RAM | ≥ 16 GB（建議 32GB） | |

> **GPU 記憶體提醒**：8GB GPU 上 Ollama 載入 llama3（~7.5GB）後幾乎佔滿，**無法同時**跑
> GPU 訓練；本機測訓練時需先停 Ollama（見 §6）。架構正式設計是 GPU 節點與服務層分離。

## 2. 一次性設定（.env 機密）

服務層 `.env` 需含三把機密（缺一即 fail-fast 無法啟動）：`JWT_SECRET_KEY`、`WORKER_API_TOKEN`、`SECRETS_MASTER_KEY`。
本機已存在 `.env`（皆已設定）。若要重新產生：

```bash
python scripts/setup_env.py        # 互動式產生 .env（服務層 / GPU 分機 模式）
```

雲端 API key（`ANTHROPIC_API_KEY` / `OPENAI_API_KEY` / `GOOGLE_API_KEY`）目前留空 → 雲端模型預設不公開，
平台只用本機 Ollama。日後接 key 後在 admin 管理頁把對應模型設「公開」即可。

## 3. 完整啟動步驟

```bash
cd CodeSpace

# (1) 核心層：nginx(:80,:8888) + job-scheduler(:8002)  ── 會 build scheduler image
docker compose up -d

# (2) AI 模型層：ollama + portkey + open-webui
#     有 NVIDIA GPU → 疊加 GPU override 讓 Ollama 用 GPU：
docker compose -f docker-compose.ai-models.yml -f docker-compose.ai-models.gpu.yml up -d
#     無 GPU → 只用基礎檔（CPU）：
#     docker compose -f docker-compose.ai-models.yml up -d

# (3) 下載聊天/簡報用的本機模型（seed 預設模型對應 llama3:latest）
docker exec ai-platform-ollama ollama pull llama3:latest

# (4)（選用，需 GPU）GPU 訓練 Worker
cd gpu-worker
#   .env 需確認：SERVICE_LAYER_URL=http://host.docker.internal:8002（同機 Docker Desktop）
#               API_TOKEN 與服務層 .env 的 WORKER_API_TOKEN 一致
#               單機共用 GPU：GPU_IDLE_UTIL_THRESHOLD=90（見 §6）
docker compose up -d --build
cd ..
```

啟動後容器應有：`ai-platform-nginx`、`ai-platform-scheduler`、`ai-platform-ollama`、
`ai-platform-portkey`、`ai-platform-webui`、（選用）`mcu-gpu-worker`。

|---|---|---|
| 使用者平台 | http://localhost/train/ | 學生/教師主介面 |
| 管理員平台 | http://localhost:8888/ | admin（本機帳號 `admin`）|

## 4. 部署中發現並修正的問題（已寫回設定/程式）

| # | 問題 | 修正 |
|---|---|---|
| 1 | `/api/v1/models` 回 502（nginx 漏掛該前綴 location，落到 Open WebUI） | `infrastructure/nginx.conf` 補 `^/api/v1/models` location |
| 2 | Portkey OSS gateway **實際聽 8787**（`PORT` env 無效），compose 卻對 8000 | compose `8787:8787`；`.env` `PORTKEY_URL` 與 `config.py` 預設改 8787；open-webui `OLLAMA_API_BASE_URL` 改 8787 |
| 3 | Portkey OSS 為 **header 路由**（忽略 mount 的 config.yaml），Ollama 無 base URL → 500 | `chat.py` 對 ollama provider 加 `x-portkey-custom-host`；`config.py` 加 `OLLAMA_BASE_URL` |
| 4 | gpu-worker `SERVICE_LAYER_URL` 壞值（`http://http:127.0.0.1:8002`，且容器內 127.0.0.1≠主機） | 改 `http://host.docker.internal:8002` |
| 5 | 桌機 GPU 被 OS/瀏覽器/Ollama 佔用，util 永遠 >10% → Worker 視為不空閒、不派工 | worker 加 `GPU_IDLE_UTIL_THRESHOLD` env（預設 10；單機設 90） |
| 6 | 兄弟容器模式：worker 把 `run.sh` 寫進**自己容器**的 `/tmp`，主機 docker `-v` 看不到 → `run.sh not found` | worker 改用 `bash -euc <inline_code>` 直接執行，不再掛載暫存檔（跨平台可靠） |

> Ollama GPU 加速採**選用 override 檔** `docker-compose.ai-models.gpu.yml`（基礎檔維持 CPU 可攜）。

## 6. 單機 GPU 注意事項（重要）

8GB GPU 上 **Ollama 與訓練無法同時用 GPU**：
- 跑聊天/簡報 → Ollama 佔 GPU（正常）。
- 要測/跑 **GPU 訓練** → 先釋放 GPU：`docker stop ai-platform-ollama`，跑完再 `docker start ai-platform-ollama`。
- 桌機 GPU 還會被 OS/瀏覽器佔用，故 worker 的「空閒判定」門檻已可調：`gpu-worker/.env` 的
  `GPU_IDLE_UTIL_THRESHOLD`（單機設 90；**專用 GPU 節點請保持 10**）。

正式環境建議：GPU 訓練放**獨立 GPU 節點**（headless，GPU 真正閒置），服務層只跑推理。

## 8. 疑難排解

| 症狀 | 可能原因 / 處置 |
|---|---|
| `/api/v1/...` 回 502 | 對應後端容器未起 / nginx 缺該前綴 location（見 §4-#1）|
| 聊天無回應、scheduler log 顯示 Portkey 連線失敗 | Portkey 未起或 port 不對（須 8787，見 §4-#2/#3）；`ollama list` 確認 llama3 已 pull |
| 模型下拉空白 | models 表無公開且適用該工具的模型；用 admin 新增/公開，或確認 seed 生效 |
| GPU 訓練一直 pending | GPU util ≥ 門檻（Ollama/桌機佔用）→ 停 Ollama 或調高 `GPU_IDLE_UTIL_THRESHOLD`；worker 心跳 `gpus=[]` 即此情況 |
| 訓練 `run.sh not found` | 舊版兄弟容器掛載問題（見 §4-#6，已修為 `bash -c`）；重 build worker |
| Lab「開啟 Notebook」失敗 | 確認 `aibase/code-server:2026-spring` image 已 build、docker.sock 已掛載 |
