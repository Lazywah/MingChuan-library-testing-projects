# 00 — 本機完整部署指南（單機 PC / Windows + Docker Desktop）

> 本文記錄在**單一 PC** 上把整個平台（核心服務 + AI 模型 + GPU 訓練）完整跑起來的
> 所需工具、環境、啟動步驟、測試結果，以及部署過程中發現並修正的問題。
> 撰寫時的實測環境：**Windows 11 + Docker Desktop + NVIDIA RTX 4060（8GB）**。

## 1. 所需工具 / 環境

## 3. 完整啟動步驟（全新機器，依序執行）

> 🚀 **懶人包（Windows）**：跑完 `python scripts\setup_env.py` 與 `build-all.sh` 後，
> 直接執行 **`scripts\bootstrap.bat`** —— 它會自動依正確順序做 step (2)~(5)
> （起 AI 模型層 → pull 3 模型 → 起核心 → 建 admin → 驗證 KB）。下面是手動逐步版。

> ⚠️ 順序很重要：**先 build base images、先把 Ollama 模型 pull 好，再起核心**，否則
> ① Lab/訓練沒有 image 開不了；② 小基開機那次 KB 匯入會因模型未就緒而失敗。

```bash
cd CodeSpace

# (0) 一次性產生 .env 機密（互動式；同時寫 gpu-worker/.env）
python scripts/setup_env.py

# (1) Build base images（Lab + 各訓練環境）── ⚠️ 很耗時(可達數十分鐘)、需 vscode-extension/
#     沒有要用 Lab / GPU 訓練可暫時跳過；要用 Notebook 就一定要 build code-server
bash infrastructure/base-images/build-all.sh           # 產生 aibase/code-server、pytorch、tensorflow…
#     Windows 無 bash 時可用 Git Bash 執行，或逐一 docker build（見該腳本內容）

# (2) AI 模型層：ollama + portkey + open-webui
#     有 NVIDIA GPU → 疊加 GPU override 讓 Ollama 用 GPU：
docker compose -f docker-compose.ai-models.yml -f docker-compose.ai-models.gpu.yml up -d
#     無 GPU → 只用基礎檔（CPU，較慢）： docker compose -f docker-compose.ai-models.yml up -d

# (3) Pull 全部 3 個本機模型（缺任一都會有功能壞掉）
docker exec ai-platform-ollama ollama pull llama3:latest        # chat / 簡報
docker exec ai-platform-ollama ollama pull nomic-embed-text     # 小基 RAG 向量 (RAG_EMBED_MODEL)
docker exec ai-platform-ollama ollama pull qwen2.5:7b           # 小基 客服/家教回覆 (RAG_CHAT_MODEL)

# (4) 核心層：nginx(:80,:8888) + job-scheduler(:8002)  ── 會 build scheduler image
#     此時 Ollama+模型已就緒 → scheduler 啟動時會自動匯入知識庫 (KB)
docker compose up -d --build
#     （若 (4) 不慎早於 (2)(3) 執行 → KB 會是空的，補：docker compose restart job-scheduler，
#       或 admin token 打 POST /api/v1/assistant/reindex）

# (5) 建立第一個管理員帳號（平台預設沒有 admin）
#     Windows：直接雙擊 / 執行 scripts\create-admin.bat（密碼寫在檔內，登入後立即改）
scripts\create-admin.bat

# (6)（選用，需 GPU）GPU 訓練 Worker
cd gpu-worker
#   .env 確認：SERVICE_LAYER_URL=http://host.docker.internal:8002（同機 Docker Desktop）
#             API_TOKEN 與服務層 .env 的 WORKER_API_TOKEN 一致；單機共用 GPU：GPU_IDLE_UTIL_THRESHOLD=90（見 §6）
docker compose up -d --build
cd ..
```

啟動後容器應有：`ai-platform-nginx`、`ai-platform-scheduler`、`ai-platform-ollama`、
`ai-platform-portkey`、`ai-platform-webui`、（選用）`mcu-gpu-worker`。

> **驗證 KB 就緒**：`docker logs ai-platform-scheduler | findstr "KB status"` 應見
> `{'status': 'ok', 'chunks': 40, 'files': 8}`（不是 empty/0）。

## 4. 部署中發現並修正的問題（已寫回設定/程式）

| # | 問題 | 修正 |
|---|---|---|
| 1 | `/api/v1/models` 回 502（nginx 漏掛該前綴 location，落到 Open WebUI） | `infrastructure/nginx.conf` 補 `^/api/v1/models` location |
| 2 | Portkey OSS gateway **實際聽 8787**（`PORT` env 無效），compose 卻對 8000 | compose `8787:8787`；`.env` `PORTKEY_URL` 與 `config.py` 預設改 8787；open-webui `OLLAMA_API_BASE_URL` 改 8787 |
| 3 | Portkey OSS 為 **header 路由**（忽略 mount 的 config.yaml），Ollama 無 base URL → 500 | `chat.py` 對 ollama provider 加 `x-portkey-custom-host`；`config.py` 加 `OLLAMA_BASE_URL` |
| 4 | gpu-worker `SERVICE_LAYER_URL` 壞值（`http://http:127.0.0.1:8002`，且容器內 127.0.0.1≠主機） | 改 `http://host.docker.internal:8002` |
| 5 | 桌機 GPU 被 OS/瀏覽器/Ollama 佔用，util 永遠 >10% → Worker 視為不空閒、不派工 | worker 加 `GPU_IDLE_UTIL_THRESHOLD` env（預設 10；單機設 90） |
| 6 | 兄弟容器模式：worker 把 `run.sh` 寫進**自己容器**的 `/tmp`，主機 docker `-v` 看不到 → `run.sh not found` | worker 改用 `bash -euc <inline_code>` 直接執行，不再掛載暫存檔（跨平台可靠） |

> Ollama GPU 加速採**選用 override 檔** `docker-compose.ai-models.gpu.yml`（基礎檔維持 CPU 可攜）。

## 6. 單機 GPU 注意事項（重要）

8GB GPU 上 **Ollama 與訓練無法同時用 GPU**：
- 跑聊天/簡報 → Ollama 佔 GPU（正常）。
- 要測/跑 **GPU 訓練** → 先釋放 GPU：`docker stop ai-platform-ollama`，跑完再 `docker start ai-platform-ollama`。
- 桌機 GPU 還會被 OS/瀏覽器佔用，故 worker 的「空閒判定」門檻已可調：`gpu-worker/.env` 的
  `GPU_IDLE_UTIL_THRESHOLD`（單機設 90；**專用 GPU 節點請保持 10**）。

正式環境建議：GPU 訓練放**獨立 GPU 節點**（headless，GPU 真正閒置），服務層只跑推理。

## 6.1 `mcu-gpu-worker` 的生命週期（獨立節點，平時不需手動關閉）

`mcu-gpu-worker` **刻意獨立於核心 stack**：它有自己的 compose（`gpu-worker/docker-compose.yml`）與獨立 network，模擬一台**遠端 GPU 運算節點**——正式環境會跑在另一台機器上，透過網路向服務層輪詢領取訓練任務。因此 `docker compose down`（核心）**不會**連帶停掉它。

**需要手動關閉嗎？平時不用。** 設計如下：

| 行為 | 說明 |
|---|---|
| `restart: unless-stopped` | Docker Desktop 開機時**自動拉起**，並持續存活直到你手動 `stop`。這也是為何核心 stack 已 down、它卻仍 `Up`。 |
| 核心未啟動時 | 每 `HEARTBEAT_INTERVAL`（預設 30s）對 `:8002` 心跳失敗一次，僅寫一行 WARNING log。**GPU 用量為 0**（worker 只有 `utility` 能力跑 `nvidia-smi`，本身不訓練），幾乎不耗資源。 |
| 核心恢復後 | scheduler 一起來，心跳**自動恢復**、繼續領任務，無需重啟 worker。 |

**只有這兩種情況才需要手動停：**
```bash
# (a) 長時間關閉整個平台、不想看到它持續重試的 log（暫停，Docker 重啟後仍會自己回來）
docker stop mcu-gpu-worker

# (b) 真的要它「不再自動啟動」（移除容器，需重新 up 才會回來）
cd gpu-worker && docker compose down && cd ..
```
> 注意 (a) 因 `unless-stopped`，**重啟 Docker 後 worker 會自己回來**；要徹底不自啟，請用 (b) 的 `compose down`。

> 判讀：`nvidia-smi` 的 compute-apps 清單若**只有桌面程式**（explorer / 瀏覽器 / Wallpaper Engine…）而無 Ollama / 訓練容器，代表平台層 GPU 負載為 0，顯存佔用是桌面 app，與 worker 是否在跑無關。

## 6.2 外部 AI 分流（v2.5 — 角色 gate + 帳號對應表）

自家 AI 助手成熟前，可暫時以合作廠商（如 myai168）替代給**非 admin 使用者**。廠商無 API/SSO、不可內嵌（`X-Frame-Options: SAMEORIGIN` + CSP `frame-ancestors 'self'`），故採「**導流 + 帳號對應表**」：

- **角色分流**：只有 `admin` 角色看得到內部 AI hub（chat/簡報，用於繼續開發）；其餘角色點「AI 助手」分頁 → 顯示**外部中介頁**（指派帳號 + 「前往」開新分頁；明示對話不在本平台留存）。
- **帳號對應**：平台帳號 ↔ 廠商帳號為兩套、無法 SSO 打通；admin 在廠商後台造冊後，於 admin-ui 維護對應。**只存廠商帳號名，絕不存廠商密碼**。
- **總開關（回退）**：`system_config.external_ai_url`。**空字串 = 未啟用**，使用者端中介頁顯示「即將開放」；自家 AI 成熟後清空即切回，內部 chat 程式碼全程不動。

**管理流程（admin-ui → 右側 Side Bar「外部 AI」分頁）：**
1. **設定網址**：填入廠商平台網址（如 `https://www.myai168.com/tw/ai/`）→ 儲存。留空 = 停用。
2. **造冊匯入**：「CSV 批次匯入」貼上 `platform_username,vendor_username`（每行一筆，可含表頭，重複者更新）→ 匯入。或用「新增單筆對應」。
3. **維護**：對應表可「改帳號 / 停用·啟用 / 刪除」；停用者使用者端顯示「請聯絡管理員」。

> 端點：使用者 `GET /api/v1/external-ai/me`；管理 `/api/v1/external-ai/admin/{url,accounts,import}`（皆 `require_admin`）。新表 `external_ai_accounts` 由 `init_db()` 自動建立；`external_ai_url` 由 seed 預設 MYAI 網址（可於管理頁清空回退）。

### 6.2.1 廠商 Token 同步 + Email 綁定（v2.8）

在「導流 + 對應表」之上，v2.8 補上**唯讀同步**與**顯式 email 綁定**：

- **headless 同步（唯讀）**：以管理者帳密（`.env` 的 `MYAI_ADMIN_EMAIL` / `MYAI_ADMIN_PASSWORD`）headless 登入 myai168 後台 → 匯出使用者清單（含「點數」= Token 餘額）→ 存進 `myai_accounts`。**全程唯讀，絕不回寫廠商**。可「立即同步」，或由排程每 `MYAI_SYNC_INTERVAL_HOURS` 小時（預設 6；設 0 關閉）自動跑。
  - ⚠️ **登入眉角**：myai 有 `form-action 'self'` 防跨站，登入 POST 必須帶正確 `Referer`/`Origin` 標頭，否則回 200 卻不發 session cookie（見 `services/myai_sync.py`）。
- **Email 綁定 / auto-match**：以 email 對應「平台使用者 ↔ myai 帳號」，存於 `external_ai_accounts`（`vendor_username` = myai email，`myai_vendor_sn` = 廠商穩定鍵）。同步後自動配對；admin 也可在「外部 AI」分頁「一鍵 Email 配對」、查看「未配對清單」、手動綁/解綁。**只寫本平台 DB**。
- **使用者端 Token 餘額**：AI 助手中介頁顯示「AI 點數餘額」（有綁定且同步到資料才顯示）。
- 端點：`/admin/{sync-myai,myai-accounts,auto-match,bindings,unmatched}`（皆 `require_admin`）。

### 6.2.2 跳轉登入 + 共用機台換手安全（v2.8）

**為何沒有「真・帳密自動登入」**：myai 登入要 email + 密碼，我們**只存帳號、絕不存密碼**（資安原則）。要替學生自動送出密碼就得逐一保管其密碼 → 不做。OAuth 僅供 AI 推論（scope `mcp`），非使用者 SSO。故自動化只到「**導流 + 顯示該用哪個 email + 提醒**」，密碼由學生自行輸入或瀏覽器密碼管理員處理。

**共用機台（圖書館／實驗室多人輪用同一台）務必注意換手**，否則下一位可能延續上一位的登入：

1. **平台端**：AI 助手中介頁提供「**結束使用 / 換人**」按鈕 → ① 開廠商登出網址殺掉 myai session ② 登出本平台並清除 `ai_hud_token`（localStorage + cookie）③ 提醒關閉 MYAI 分頁。
2. **廠商登出端點（已驗證並接上）**：`https://www.myai168.com/mcu/ai/user/logout_info`（GET 導向即清 session；實測登出前 export 200、登出後 401）。存於 `system_config.external_ai_logout_url`，admin 可在「外部 AI」分頁的「廠商登出網址」欄調整；`/external-ai/me` 會回傳給前端按鈕使用。
3. **瀏覽器端（最終保證）**：共用機仍建議以**無痕／訪客**開啟，或設定「關閉時清除 Cookie」，並**關閉「儲存密碼」**。即使有登出端點，這是防止任何 session/快取殘留給下一位的最後一道保險。
4. **桌面捷徑**：`scripts/建立桌面捷徑.bat`（改好 `PLATFORM_URL` 後執行）會在桌面建立指向**本平台登入頁**的捷徑 —— 不嵌任何帳密，每位學生自己登入。

## 6.3 RAG 客服/導覽助手「小基」（v2.6 — 右下角浮動泡泡）

獨立於主「AI 助手」入口的浮動客服泡泡，以 RAG（`job-scheduler/knowledge/*.md` + 本機 Ollama）回答平台使用問題。`/api/v1/assistant/ask` **公開、不扣 Token、限流 30/分、直接打 Ollama 不經 Portkey**。

⚠️ **三個「不做會無聲失敗」的部署前置（最容易漏）：**

1. **rebuild scheduler image（不是只 restart！）**——知識庫是 `Dockerfile: COPY knowledge/ /app/knowledge/`（build 時打包），compose **未掛載** `knowledge/`。若 image 在 `.md` 出現前 build，容器內 `/app/knowledge` 會是空的（log 顯示 `KB status: empty`）。
   ```bash
   docker compose up -d --build job-scheduler   # 把 knowledge/*.md 打進 image
   ```
   > 開發時若想免 rebuild 即時更新 KB，可在 compose 的 job-scheduler `volumes:` 加 `- ./job-scheduler/knowledge:/app/knowledge:ro`（與 app/ 熱掛載同理）。
2. **Ollama 必須在跑，且 pull 兩個模型**（否則客服不回話、看似壞掉）：
   ```bash
   docker compose -f docker-compose.ai-models.yml -f docker-compose.ai-models.gpu.yml up -d
   docker exec ai-platform-ollama ollama pull nomic-embed-text   # RAG_EMBED_MODEL（向量）
   docker exec ai-platform-ollama ollama pull qwen2.5:7b          # RAG_CHAT_MODEL（回覆）
   ```
3. **nginx 已補 `/api/v1/assistant` location**（v2.6 整合修正）——`/ask` 是 SSE，須關 buffering；缺此 location 時 widget 經 :80 會 502（同 `/api/v1/models` 類問題）。已內建於 `nginx.conf`，**改 conf 後記得 `nginx -s reload`**。

**驗證 / 維運：**
```bash
docker exec ai-platform-scheduler ... # 重啟後看 log：KB status: {'status':'ready','chunks':N}
curl -N -X POST -H "Content-Type: application/json" -d '{"question":"如何登入"}' http://localhost/api/v1/assistant/ask   # 應回 200 text/event-stream
# admin 重建知識庫（改 knowledge/*.md 後）：POST /api/v1/assistant/reindex
```
> 測試：`tests/test_rag_service.py`（9 項純邏輯）；`job-scheduler/.venv` 不完整（缺 httpx），請用系統 Python 3.13 跑。

### v2.7 程式家教模式（Notebook 程式輔導）

小基新增第二模式，右下角泡泡可切換「客服 ⇄ 程式家教」：

- **guide（客服）**：維持原狀（公開、不需登入、不扣 token）。
- **code（程式家教）**：`/ask` 帶 `mode:"code"`，**需登入**（帶 `Authorization: Bearer`）；用 qwen2.5:7b + 家教式提示。可附使用者「自己」Lab 裡的檔：`GET /api/v1/assistant/lab-files` 列檔 → `/ask` 帶 `file_path` → 後端用 Docker `get_archive` 讀 `cs-<uid>:/home/coder/<file_path>`。

## 8. 疑難排解

| 症狀 | 可能原因 / 處置 |
|---|---|
| `/api/v1/...` 回 502 | 對應後端容器未起 / nginx 缺該前綴 location（見 §4-#1）|
| 聊天無回應、scheduler log 顯示 Portkey 連線失敗 | Portkey 未起或 port 不對（須 8787，見 §4-#2/#3）；`ollama list` 確認 llama3 已 pull |
| 模型下拉空白 | models 表無公開且適用該工具的模型；用 admin 新增/公開，或確認 seed 生效 |
| GPU 訓練一直 pending | GPU util ≥ 門檻（Ollama/桌機佔用）→ 停 Ollama 或調高 `GPU_IDLE_UTIL_THRESHOLD`；worker 心跳 `gpus=[]` 即此情況 |
| 訓練 `run.sh not found` | 舊版兄弟容器掛載問題（見 §4-#6，已修為 `bash -c`）；重 build worker |
| Lab「開啟 Notebook」失敗 | 確認 `aibase/code-server:2026-spring` image 已 build、docker.sock 已掛載 |
