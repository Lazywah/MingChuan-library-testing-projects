# AI 訓練平台 - 系統層級運行流程與連線指南

本平台採用嚴謹的**三層分離架構**，每一層皆有獨立的作業系統與職責。以下將詳細說明各層級在實際運行時的互動流程，以及「服務層」如何安全地連線並調度「GPU 高階伺服器」。

---

## 🏗️ 1. 各層級運行流程 (Operational Flow)

### 💻 第一層：工作站 (Workstation)
*   **環境**：使用者的筆電、PC，無需安裝任何系統套件，僅需瀏覽器。
*   **運行流程**：
    1.  使用者打開瀏覽器，輸入服務層的 IP (如 `http://192.168.1.10/train/`)。
    2.  瀏覽器從**服務層 (Nginx)** 下載靜態的 HTML / CSS / JS 檔案。
    3.  使用者在網頁上點擊「登入」或「提交訓練任務」時，瀏覽器會將使用者的操作轉化為 RESTful API 請求，並發送給服務層。
    4.  瀏覽器負責維持長連線 (如 SSE)，即時接收並渲染從服務層回傳的模型對話結果或訓練進度條。

### ☁️ 第二層：服務層 (Service Layer)
*   **環境**：Ubuntu 22.04 / 24.04 LTS，運行 Docker Engine。
*   **角色**：平台的大腦與中樞網關。
*   **運行流程**：
    1.  **Nginx (Port 80)** 接收來自工作站的請求，區分靜態資源 `/train/` 與 API `/api/v1/`。
    2.  **Job Scheduler (FastAPI)** 接收 API 請求。若是「登入」則比對 SQLite 資料庫；若是「AI 對話」則將請求轉發至內部的 Portkey 容器。
    3.  若收到「訓練任務」，Scheduler 會將任務存入 SQLite，狀態標為 `pending`。
    4.  背景排程迴圈 (Background Loop) 每數秒喚醒一次，發現有 `pending` 任務時，便主動發起 SSH 連線至第三層（GPU 高階伺服器）查詢資源。

### 🚀 第三層：GPU高階伺服器 (GPU High-End Server)
*   **環境**：Windows 11，配備高階 NVIDIA 顯示卡，並開啟 OpenSSH Server。
*   **角色**：純粹的「算力勞工」。
*   **運行流程**：
    1.  平時處於待命狀態，OpenSSH 服務 (Port 22 或 2222) 持續監聽來自服務層的指令。
    2.  當服務層透過 SSH 派發 Python 訓練指令時，Windows 11 會喚起 CUDA 資源並開始執行訓練腳本 (`nohup python ...` 或 `Start-Process`)。
    3.  訓練過程中，產生的日誌 (logs) 或進度檔會被寫入本地磁碟（或網路共享磁碟）。服務層會定期透過 SSH 讀取這些檔案來更新前端進度。
    4.  訓練結束，釋放 GPU 資源，等待服務層派發下一個任務。

## 🔗 2. 服務層與 GPU 高階伺服器的連線步驟

由於服務層 (Ubuntu) 必須「無密碼、無人工介入」地遠端操控 GPU 伺服器 (Windows 11)，雙方依賴 **SSH RSA 金鑰認證** 進行通訊。

### 步驟一：在服務層 (Ubuntu) 生成 SSH 金鑰
服務層是「發起連線」的一方，因此必須在服務層產生金鑰。
1. 登入服務層 (Ubuntu) 終端機。
2. 執行指令生成金鑰 (不設定密碼)：
   ```bash
   ssh-keygen -t rsa -b 4096 -f ~/.ssh/gpu_key -N ""
   ```
3. 產生後，會有兩個檔案：
   * `~/.ssh/gpu_key` (私鑰，**絕對不可外洩**，留在 Ubuntu)
   * `~/.ssh/gpu_key.pub` (公鑰，需要丟給 Windows 11)
4. 查看並複製公鑰內容：
   ```bash
   cat ~/.ssh/gpu_key.pub
   ```

### 步驟二：配置 GPU 伺服器 (Windows 11) 接收連線
Windows 11 是「被連線」的一方，必須安裝 SSH 服務並信任剛剛的公鑰。
1. 在 Windows 11 上，以系統管理員身分執行初始化腳本（包含啟用 OpenSSH Server）：
   ```powershell
   cd C:\Users\gpu_admin\Desktop\gpu-setup
   .\setup.ps1
   ```
2. 建立 authorized_keys 檔案並貼上公鑰：
   打開 PowerShell，進入使用者目錄下的 `.ssh` 資料夾：
   ```powershell
   mkdir $env:USERPROFILE\.ssh -Force
   notepad $env:USERPROFILE\.ssh\authorized_keys
   ```
   將步驟一中 `cat` 出來的 `ssh-rsa AAAA...` 內容貼入記事本並存檔。
3. (選用，但強烈建議) 執行安全強化腳本，關閉密碼登入：
   ```powershell
   .\ssh-hardening.ps1
   ```

### 步驟三：在服務層測試連線
回到服務層 (Ubuntu)，手動測試是否能無密碼登入 Windows 11：
```bash
# 假設 Windows 11 IP 為 192.168.1.100，使用者名稱為 gpu_admin
ssh -i ~/.ssh/gpu_key gpu_admin@192.168.1.100 "nvidia-smi"
```
如果畫面成功印出 Windows 11 上的 NVIDIA 顯示卡狀態，且**沒有詢問密碼**，代表連線通道已打通！

### 步驟四：告知系統排程器 (Job Scheduler)
最後，必須將這條打通的連線資訊寫入系統設定，讓程式自動接手。
在服務層編輯 `job-scheduler/app/scheduler_policy.yaml`：

```yaml
mock_mode: false    # 關閉模擬模式，啟用真實連線

nodes:
  - id: "win11-gpu-node-01"
    host: "192.168.1.100"       # GPU 高階伺服器 (Windows 11) 的 IP
    port: 22                     # SSH Port (若執行過 hardening 腳本，請確認 Port 號)
    username: "gpu_admin"        # Windows 11 的登入帳號
    ssh_key_path: "/root/.ssh/gpu_key"  # 服務層中私鑰的絕對路徑
```
儲存後，重啟服務層的容器：
```bash
docker compose restart job-scheduler
```

完成以上步驟，當工作站的使用者按下「提交任務」時，服務層就會自動透過金鑰連入 Windows 11，喚起 GPU 開始訓練。
