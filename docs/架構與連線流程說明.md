# AI 訓練平台 - 系統層級運行流程與連線指南

本平台採用嚴謹的**三層分離架構**，每一層皆有獨立的作業系統與職責。以下將詳細說明各層級在實際運行時的互動流程，以及「服務層」如何安全地連線並調度「GPU 高階伺服器」。

---

## 🏗️ 1. 各層級運行流程 (Operational Flow)

### 💻 第一層：工作站 (Workstation)
*   **環境**：使用者的筆電、PC，無需安裝任何系統套件，僅需瀏覽器。
*   **運行流程**：
    1.  使用者打開瀏覽器，輸入服務層的 IP (如 `http://192.168.1.10/train/`)。
    2.  瀏覽器從**服務層 (Nginx)** 下載靜態的 HTML / CSS / JS 檔案。
    3.  使用者在網頁上點擊「登入」或「提交訓練任務」時，瀏覽器會將使用者的操作轉化為 RESTful API 請求，並發送給服務層。
    4.  瀏覽器負責維持長連線 (如 SSE)，即時接收並渲染從服務層回傳的模型對話結果或訓練進度條。

### ☁️ 第二層：服務層 (Service Layer)
*   **環境**：Ubuntu 22.04 / 24.04 LTS，運行 Docker Engine。
*   **角色**：平台的大腦與中樞網關。
*   **運行流程**：
    1.  **Nginx (Port 80)** 接收來自工作站的請求，區分靜態資源 `/train/` 與 API `/api/v1/`。
    2.  **Job Scheduler (FastAPI)** 接收 API 請求。若是「登入」則比對 SQLite 資料庫；若是「AI 對話」則將請求轉發至內部的 Portkey 容器。
    3.  若收到「訓練任務」，Scheduler 會將任務存入 SQLite，狀態標為 `pending`。
    4.  開放 `POST /api/v1/worker/take` 與 `POST /api/v1/worker/jobs/{job_id}/update` 端點，被動等待 GPU 伺服器的 Worker Agent 來認領任務並更新狀態。

### 🚀 第三層：GPU高階伺服器 (GPU High-End Server)
*   **環境**：Windows 11，配備高階 NVIDIA 顯示卡，並運行 WSL 2 與 Docker Desktop。
*   **角色**：純粹的「算力勞工」。
*   **運行流程**：
    1.  平時處於待命狀態，**Worker Agent (Python)** 在背景定期（如每 5 秒）使用 `nvidia-smi` 檢查本機 GPU 負載。
    2.  當發現 GPU 空閒時，主動發起 HTTP 請求向服務層索取 `pending` 的任務。
    3.  領取任務後，Worker 在本地動態啟動 `docker run --gpus all` 指令，將訓練作業丟入隔離的 PyTorch 容器中執行。
    4.  訓練過程中，Worker 即時捕捉容器的 stdout 日誌與進度，並透過 HTTP 傳送給服務層更新前端進度。
    5.  訓練結束，銷毀容器 (`--rm`)，釋放 GPU 資源，等待領取下一個任務。

## 🔗 2. 服務層與 GPU 高階伺服器的連線步驟

由於服務層 (Ubuntu) 必須將任務安全地交給 GPU 伺服器 (Windows 11) 執行，雙方採用 **HTTP API Polling (主動輪詢)** 機制進行通訊。

### 步驟一：啟動服務層 API
服務層啟動後，會暴露一組 `Worker` 專用的 RESTful API：
* `POST /api/v1/worker/take`: 供 GPU 伺服器領取任務。
* `POST /api/v1/worker/jobs/{job_id}/update`: 供 GPU 伺服器回報進度與日誌。

### 步驟二：配置 GPU 伺服器 (Worker Agent)
GPU 伺服器是「發起連線」的一方，必須知道主機的位址與驗證 Token。
1. 在 Windows 11 上，準備好 `gpu-worker` 資料夾。
2. 編輯 `docker-compose.yml`，設定 `SERVICE_LAYER_URL` 指向服務層主機的 IP。
3. 設定相同的 `API_TOKEN` 以通過服務層的簡單驗證。
4. 執行 `docker-compose up -d`，Worker Agent 就會開始定期呼叫主機的 `/take` API。

### 步驟三：掛載共享儲存區
由於訓練腳本與資料集存放於服務層，GPU 伺服器必須透過 NFS 或 SMB 協定，將服務層的儲存目錄掛載到本機 (如 `C:\storage`)。
Worker 在收到任務後，會透過 Docker Volume (`-v C:\storage:/workspace`) 將這份共享目錄直接餵給訓練容器使用。

完成以上步驟，當工作站的使用者按下「提交任務」時，服務層會將任務存入資料庫；GPU 伺服器的 Worker 會在幾秒內偵測到新任務，並主動拉回本地執行。

## 🛡️ 3. 資安優勢與防護建議 (Security Advantages & Considerations)

本平台捨棄了傳統的 SSH 推播 (Push) 架構，改用 Worker Agent (Pull) 架構，這在根本上解決了大部分的資安疑慮：

### ✅ 優勢一：無需開啟對外 Port (NAT Friendly)
* **舊版痛點**：GPU 伺服器必須開啟 SSH Port 供外部連入，面臨暴力破解與網路掃描風險。
* **新版架構**：連線是由 GPU 伺服器**主動由內向外**發起 HTTP 請求。因此，GPU 伺服器可以安全地藏在校園防火牆或 NAT 後方，**完全不需開啟任何對外 Port**。

### ✅ 優勢二：阻斷橫向移動風險
* **舊版痛點**：服務層保管了存取 GPU 伺服器的最高權限 SSH 私鑰。若服務層遭駭，駭客可直接用該私鑰免密登入 GPU 伺服器，造成災情擴散。
* **新版架構**：服務層**沒有**控制 GPU 伺服器的能力。即使服務層遭駭，駭客頂多只能在 API 佇列中塞入惡意任務。但惡意任務會在 GPU 伺服器受限的 Docker 容器內執行，無法破壞 Windows 11 系統本身。

### ✅ 優勢三：沙盒隔離與環境淨空 (Dockerization)
* **舊版痛點**：訓練腳本直接運行在 Windows 11 裸機上，可能遺留垃圾檔案或發生套件版本衝突。
* **新版架構**：所有的訓練腳本都是在臨時啟動的 Linux 容器 (`docker run --rm`) 內執行。無論腳本寫得多糟，甚至是惡意腳本，容器銷毀後一切歸零，保證了 GPU 伺服器的絕對乾淨與安全。

### 🚨 注意事項：API 防護與存取控制
雖然 Pull 架構極大提升了 GPU 端的安全性，但服務層 (主機端) 的防護仍需留意：
1. **API Token 保護**：在 `gpu-worker` 與主機之間設定的 `API_TOKEN` 是防範惡意 Worker 來亂領任務或亂傳假進度的關鍵。請確保該 Token 具備足夠長度並定期更換。
2. **防火牆白名單 (IP Whitelisting)**：在服務層 (Ubuntu) 的 Nginx 或 UFW 設定中，建議將 `/api/v1/worker/*` 路由的存取權限，**嚴格限制為只允許已知的 GPU 伺服器 IP** 連入，進一步降低遭外部攻擊的風險。
